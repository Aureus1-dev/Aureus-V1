## Experience Architecture vs. Execution Architecture

Two different things live in this repository and must not be conflated:

- **Experience Architecture** — the Canon. The Gate, the Question, the Bench, the Greeting, the Watchkeeper, the Clearing, Memory rights, the Leaving, the Quiet Day, the Homecoming, and any other named "room" are experience-design concepts. They describe how a member should *feel* moving through Aureus. They live in `LAUNCH-001-First-Members.md` and the Experience Canon — not here.
- **Execution Architecture** — this document. Work orders here are organized by **engineering Gate** (A–F, a sequencing/readiness concept) and describe **observable behavior, member outcomes, accessibility, resilience, and production readiness** — never by which Experience Canon room they happen to touch.

A single work order may (and usually will) implement pieces of several Canon rooms at once. That is expected and correct. Do not create a work order titled "Build the Greeting" or "Build the Bench" — describe instead what a member can observably do, safely and accessibly, and to what production standard.

**B5 implementation note (Authenticated and unauthenticated behavior is correct):** This work order was primarily verification, not new code — `AuthGate` (`apps/web/design-system/components/auth/AuthGate.tsx`) already correctly handles every static state (restoring, authenticated, unauthenticated, session-expired), each already covered by an existing test. **B6 implementation note (Arrival is safely interruptible and resumable):** Before this work order, `FirstRunWelcome`'s current step (`consent` → `preferences` → `hospitality` → `immediate-need` → `first-mission` → `opportunities` → `review-approval` → `next-step`) lived only in `useState`, so any reload restarted the whole wizard from `consent` — repeating already-completed steps, exactly what this work order exists to prevent. Worse, a genuine bug: once the member's immediate need created a Goal, `WelcomeFlow`'s existing `isReturningMember` check (`goals.length > 0`) would treat them as a "returning member" and redirect them to `/home` on reload, silently skipping the rest of onboarding (Opportunities, Review & Approval, Next Step) they'd never actually seen. New `arrival-progress.ts` persists the current step to `localStorage` (device-scoped, matching consent/preferences); `FirstRunWelcome` now initializes from and writes to it via a `goToStep` wrapper, and `WelcomeFlow`'s returning-member redirect now checks for a persisted incomplete step and defers to resuming the wizard instead of redirecting home when one exists. `forceNewMission` (a returning member deliberately starting a new mission) clears any stale persisted step, since that is an intentional fresh start, not a resume. Reaching `next-step` clears the persisted step (nothing left to resume). A resumed `first-mission` with no created goal and nothing in flight falls back to `immediate-need` rather than rendering a stuck blank screen. 5 new tests (mid-wizard resume, final-step clears the marker, the `WelcomeFlow`-level no-longer-redirected-home case, the first-mission stuck-screen fallback, `forceNewMission` unaffected by the existing test). No backend or schema change.

**B8 implementation note (Arrival fails safely):** Most arrival steps already handled failure honestly (`ConsentStep`, `FirstMissionStep` — retryable errors with calm, action-oriented copy from `domain-error-copy.ts`), but a real gap existed in `WelcomeFlow.tsx`: if `journey.loadGoals()` failed (network/session/service failure), the component only checked `isLoadingGoals`, never `error` — silently falling through to render the full `FirstRunWelcome` onboarding wizard as if the member were brand-new, when in fact we simply couldn't tell whether they were a returning member. That is exactly the honesty failure B8 exists to close (implying "you have no history here" when the truth is "we couldn't check").
