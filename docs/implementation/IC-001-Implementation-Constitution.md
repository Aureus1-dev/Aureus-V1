IC-001 — Implementation Constitution

Canonical Designation: IC-001

Title: Implementation Constitution

Status: Canonical

Authority: This document governs the implementation of the Aureus software platform. It is subordinate to the Aureus constitutional documents (OAS Series) and the Product Constitution. Where conflict exists, the OAS documents and Product Constitution take precedence.

---

Article I — Purpose

The purpose of this Constitution is to establish the permanent rules governing software implementation for Aureus.

It defines how the platform is built, how engineering decisions are executed, and how implementation work is performed.

This document exists to ensure that Aureus is built consistently, safely, transparently, and maintainably over the lifetime of the institution.

---

Article II — Scope

This Constitution applies to:

- Cursor
- AI coding assistants
- Human engineers
- Contractors
- Contributors
- Automated implementation systems

Every contributor shall follow these rules unless an authorized constitutional amendment provides otherwise.

---

Article III — Source of Truth

Implementation shall follow this order of authority:

1. Aureus Constitutional Documents (OAS Series)
2. Product Constitution
3. Canonical Architecture
4. Architecture Decision Records (ADRs)
5. Canonical Data Models
6. Approved Feature Specifications
7. Approved Work Orders
8. Engineering Standards
9. This Implementation Canon

No lower-authority document may contradict a higher-authority document.

---

Article IV — Implementation Principles

Every implementation shall:

- faithfully implement approved specifications;
- preserve architectural integrity;
- maintain security by default;
- prioritize correctness over speed;
- remain understandable and maintainable;
- be production-ready upon completion unless a Work Order explicitly states otherwise.

---

Article V — Prohibited Actions

No implementation shall:

- invent product requirements;
- redesign approved functionality;
- remove approved functionality without authorization;
- expand project scope without approval;
- violate architectural boundaries;
- bypass security requirements;
- intentionally reduce code quality;
- introduce undocumented behavior.

When uncertainty exists, implementation shall stop and request clarification.

---

Article VI — Work Orders

All implementation shall be executed through approved Work Orders.

Each Work Order shall define:

- objective;
- dependencies;
- scope;
- deliverables;
- acceptance criteria;
- definition of completion.

Implementation outside an approved Work Order is prohibited unless explicitly authorized.

---

Article VII — Engineering Standards

Every implementation shall conform to the Engineering Standards.

These standards govern:

- project structure;
- naming conventions;
- documentation;
- testing;
- logging;
- validation;
- security;
- dependency management;
- code quality;
- maintainability.

---

Article VIII — Testing

No implementation shall be considered complete without satisfying the testing requirements established by the applicable Work Order and Engineering Standards.

Testing shall be appropriate to the level of risk introduced by the implementation.

---

Article IX — Change Control

Changes to approved architecture, APIs, schemas, workflows, or core behavior shall follow the established change-control process.

Implementation shall not silently alter canonical behavior.

---

Article X — Definition of Done

A Work Order is complete only when:

- all acceptance criteria are satisfied;
- required tests pass;
- documentation is updated;
- architectural compliance is maintained;
- implementation is production-ready unless otherwise specified.

Partial completion shall not be represented as complete.

---

Article XI — AI Implementation Conduct

AI implementation systems shall:

- follow approved specifications exactly;
- avoid speculation;
- avoid unnecessary commentary;
- produce deterministic, reproducible implementation whenever practical;
- report blockers instead of inventing solutions outside the approved scope.

---

Article XII — Constitutional Stability

This Constitution is intended to provide long-term continuity for the Aureus engineering process.

Future improvements should strengthen clarity, quality, and maintainability while preserving compatibility with higher-authority governing documents.

No provision of this Constitution may supersede the Aureus constitutional framework or the Product Constitution.
