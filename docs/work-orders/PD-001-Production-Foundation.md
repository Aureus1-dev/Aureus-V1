# PD-001 — Production Foundation: Domain Readiness Report

**Status: Complete.** Per the Founder's explicit expansion instruction, PD-001 was broadened from the PD-000 audit's original narrow scope ("Email & Notification Delivery Production Configuration") into the complete **Production Foundation** domain — absorbing what PD-000 had originally split across PD-001 (email) and PD-004 (auth hardening), plus the rate-limiting/config-cleanup slices of PD-005, into a single domain and a single PR, per instruction: *"Do not split these into multiple production domains unless there is a technical dependency that makes it impossible to complete them together."*

No such technical-dependency split was needed. All eleven named components, plus one additional item discovered during implementation, shipped together.

---

## 1. Objective

Close every foundational, launch-blocking production gap identified by PD-000 that does not require external infrastructure this sandbox cannot provision (a live Redis cluster, a real inbox, a live Docker daemon) — so that a production deploy of Aureus fails loudly at boot on misconfiguration rather than silently degrading, and so that the authentication/authorization/rate-limiting/input-handling surface is hardened to a standard consistent with handling real user credentials and real content moderation states.

## 4. A design correction made mid-domain (worth recording)

The first attempt at the authorization-verification fix (item 4 above) added an optional `caller` parameter directly to each domain's `findById`/`findByRef` **service** method and restricted non-VERIFIED content there. This was wrong and was caught before shipping: those same service methods are called internally — without a caller — by course-authoring flows (`modules.service.ts`, `lessons.service.ts`, `path-courses.service.ts` checking a course/path exists before adding content to it), AI insights (`ai/insights/insights.service.ts`), institutional memory (`ai/memory/institutional-memory.service.ts`), and Steward recommendation validation (`stewardship/recommendations/stewardship-recommendations.service.ts`). Restricting the service method would have made all five of those internal call sites silently start throwing `NotFoundException` for legitimate DRAFT/PENDING_REVIEW content, since they never pass a caller. The fix was moved to the **controller** boundary instead — a new shared `assertContentVisible()` utility, called only after the (unrestricted, unchanged) service lookup returns, using the `caller` that genuinely reflects the real HTTP requester. Internal service-to-service calls never pass through the controller and are unaffected.

## 7. Explicitly deferred (not silently dropped)

Per the Founder's instruction to name deferred items clearly:

- **Redis-backed distributed rate limiting.** The current `@nestjs/throttler` config is in-memory, correct for a single instance, but will under/over-count across horizontally-scaled instances. This requires real Redis infrastructure to test meaningfully and belongs with the infrastructure/CI-CD domain (PD-005 in the PD-000 numbering), not this one — the *application-level* per-endpoint throttle tightening (item above) was still done here since it's independent of the storage backend.
- **OAuth/SSO and CAPTCHA.** Named as adjacent "authentication hardening" candidates in the original audit but require external provider selection/product decisions (which OAuth providers, whether CAPTCHA is even wanted given the platform's relationship-first design) that are Founder-level product decisions, not engineering defaults to assume.
- **Author self-view of own DRAFT/PENDING_REVIEW content.** The authorization fix in item 4/§4 above means a content author can no longer view their own not-yet-verified Resource/Organization/Opportunity/Course/Learning Path/Knowledge Article via the public `findById`/`findByRef` endpoint (only Steward/Admin can, matching the moderator roles already used for the `update`/verification-workflow endpoints on each domain). This is a real, narrower UX gap than the vulnerability it replaces — before this domain, *anyone* unauthenticated could see this content; now, correctly, *only moderators* can, which is strictly more secure but means an author can't preview their own submission before it's reviewed. Building proper ownership-aware visibility (distinguishing "the author of this specific record" from "any moderator") is separate per-domain design work — each domain has a different ownership model (organization-owned vs. individually-authored) — and is deferred to a future domain rather than rushed here.
- **Content Security Policy (CSP) tuning for the Next.js frontend.** `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, and `Permissions-Policy` were added; a strict CSP was deliberately not attempted in this domain because Next.js's hydration relies on inline scripts that need nonce-based CSP plumbing through middleware to avoid breaking the app, and this sandbox has no live browser to verify that plumbing didn't silently break rendering. Recommended as a follow-up once a real browser-based smoke test is available.
- **Real Docker-daemon build verification.** `docker-compose.yml` was reviewed and fixed for the Joi-empty-string interaction bug, but no live Docker daemon was available in this sandbox to actually run `docker compose up --build` — consistent with the same limitation noted in the production runbook since PR-002.
- **ParseUUIDPipe consistency.** Nearly every controller across the codebase uses `@Param('id') id: string` without `ParseUUIDPipe`. Investigated and confirmed **not a security bug**: `AllExceptionsFilter` already catches `Prisma.PrismaClientValidationError` (thrown for a malformed UUID passed to a Prisma query) and returns a generic 400 "Invalid database query" — no 500, no data leak, no bypass. Adding `ParseUUIDPipe` everywhere would be a cosmetic/defense-in-depth improvement touching ~40+ controllers, including some (`opportunities.controller.ts`'s `ref` param) that are deliberately non-UUID human-readable refs, at a scale disproportionate to its actual risk. Recorded here as explicitly deferred, not overlooked.

## 8. Production risk if PD-001 had shipped incomplete

Before this domain: a production deploy could boot with `CORS_ORIGIN=*` (silently disabling credentialed CORS), no real SMTP configured (password reset/verification silently going nowhere), an AI provider selected with no API key (runtime failures instead of a boot-time signal), Swagger publicly exposing the full API schema, no second factor on any account, an unverified email able to log in indefinitely, a member able to self-verify their own email or un-suspend their own account via `PATCH /users/:id`, and unauthenticated read access to rejected/draft marketplace content. All of these are now closed or — where genuinely out of scope for this sandbox — explicitly named above rather than silently absent.

None of this domain's changes touch AI request handling, provider selection, budget enforcement, or the orchestration/institutional-memory layers built in PR-002 through PR-004 — confirmed by the full AI test suite passing unchanged and by the internal-call-site audit in §4 (which exists specifically because AI services *do* depend on the content-lookup methods this domain touched, and had to be protected from a regression).
